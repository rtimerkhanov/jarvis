## 1. Продукт

**pillar-bot** — self-hosted Telegram-бот (Node.js Docker, Render) для поддержки ежедневных опор. Работает в привязанном Telegram-канале (API key + channel ID).

## 2. Use Cases

Крупные сценарии, которые система автоматизирует. Каждый — самостоятельная ценность, можно приоритизировать и переставлять.

| # | Use Case | Что делает пользователь | Что делает система | Эпики |
|---|----------|------------------------|--------------------|-------|
| **UC-1** | **Трекинг питания** | Скидывает фото еды или пишет текстом | Распознаёт → считает КБЖУ → сохраняет → напоминает если не ел | 3, 10 (US-10.7) |
| **UC-2** | **Оцифровка блокнота** | Фотографирует бумажный блокнот (утро/вечер) | Парсит рукописный текст → сохраняет метрики, привычки, вещества → отмечает привычки в TickTick | 4, 8 |
| **UC-3** | **Дневник и разгрузка мыслей** | Пишет поток сознания или заметки с терапии | Сохраняет без комментариев → вылавливает задачи → создаёт в TickTick inbox | 5 |
| **UC-4** | **Карта дня (без слепых зон)** | Живёт обычный день | Собирает данные из обоих календарей → находит пробелы → спрашивает чем занимался → строит полный таймлайн | 6, 9 |
| **UC-5** | **Backbone рутины** | Описывает текстом идеальный распорядок | Хранит и версионирует → сравнивает план vs реальность → показывает отклонения → итерируется через дневник + Cowork | 12 |
| **UC-6** | **Ежедневные брифинги и чекпоинты** | Просыпается / обедает / ложится спать | Утренний брифинг (сон + календарь + задачи + рецепт), обеденный/вечерний чекпоинт, wind-down | 10 |
| **UC-7** | **Рецепты по контексту** | Спрашивает "что приготовить?" | Ищет в базе с учётом: КБЖУ за день, время до встречи, ротация, предпочтения | 7 |
| **UC-8** | **Еженедельная/ежемесячная аналитика** | Открывает Cowork в воскресенье | Тренды, корреляции, паттерны веществ, % привычек, аналитика времени, рекомендации, ревью backbone | 11 |

### Зависимости между UC

```
UC-1 (питание) ──────────────┐
UC-2 (блокнот) ──────────────┤
UC-3 (дневник) ──────────────┼──→ UC-6 (брифинги) ──→ UC-8 (аналитика)
UC-4 (карта дня) ────────────┤
UC-5 (backbone) ─────────────┘
UC-7 (рецепты) ──────────────┘
```

UC-1, UC-2, UC-3 — можно запускать независимо. UC-4 и UC-5 — тоже, но дают больше ценности вместе с UC-6. UC-7 — автономен, но лучше с UC-1. UC-8 — финальный слой, работает на данных всех предыдущих.

## 3. Принципы

1. **Блокнот — источник правды** для субъективных данных. Приоритет сна: блокнот > Watch.
2. **Минимум трения.** 5-15 минут/день. Бот понимает контекст — не нужно объяснять, что ты делаешь.
3. **$0 инфраструктура + ~$2/мес Claude API** для качественного распознавания.
4. **Шаблон блокнота эволюционирует.** JSONB, гибкая схема.
5. **Без оценок.** Вещества, дневник, записи с психолога — факты и пространство, не мораль.
6. **Без слепых зон.** Система знает, чем ты занят в каждый момент дня.

## 4. Типы действий бота (контекстный роутинг)

Бот должен понимать из контекста, что именно ты ему отправляешь, без команд. Вот полная классификация:

### 4.1 Матрица распознавания

| Что приходит | Как бот понимает | Действие |
|-------------|-----------------|----------|
| **Фото еды** | Фото + нет маркеров блокнота, визуально еда | → Claude Vision → КБЖУ → `nutrition_log` |
| **Фото блокнота (утро)** | Фото + виден шаблон с чекбоксами, или caption "утро" | → Claude Vision → парсинг → `notebook_entry` + TickTick |
| **Фото блокнота (вечер)** | Фото + вечерний шаблон, или caption "вечер" | → Claude Vision → парсинг → `notebook_entry` + TickTick |
| **Фото скриншота календаря** | Фото + виден UI календаря, или caption "календарь" | → Claude Vision → парсинг → `time_log` |
| **Текст о еде** | "Съел...", "на обед...", описание еды | → Claude → расчёт КБЖУ → `nutrition_log` |
| **Свободный текст (мысли)** | Нет маркеров еды/задач, длинный текст, поток сознания | → `journal_entry` (дневник) |
| **Записи с психолога** | "С психолога:", "сессия:", "терапия:", длинный текст с инсайтами | → `journal_entry` (тип: therapy) |
| **Вопрос о рецепте** | "Что приготовить", "рецепт", "на обед что" | → поиск в `recipe` → рекомендация |
| **Ответ на вопрос бота** | Бот спрашивал "чем занимался?" → короткий ответ | → `time_log` (заполнение слепой зоны) |
| **Отметка добавок** | "Добавки", "витамины", Action Button | → `supplement_log` + TickTick |
| **Описание рутины** | "Рутина:", "мой распорядок:", /routine, или текст с расписанием по часам | → `routine_config` (новая версия backbone) |
| **Непонятное** | Не удалось классифицировать | → inline-кнопки: 🍽 Еда | 📓 Блокнот | 💭 Дневник | ❓ Другое |

### 4.2 Логика роутинга (в коде)

```typescript
async function routeMessage(msg: TelegramMessage): Promise<ActionType> {
  // 1. Фото — классификация через Claude Vision
  if (msg.photo) {
    const classification = await classifyPhoto(msg.photo, msg.caption);
    // Claude Vision сам определяет: food | notebook_morning | notebook_evening | calendar_screenshot | unknown
    return classification;
  }

  // 2. Текст — классификация по паттернам + контексту
  if (msg.text) {
    // Ожидаем ответ на вопрос бота?
    if (pendingQuestion(msg.chat_id)) {
      return handlePendingAnswer(msg);
    }

    // Маркеры еды
    if (matchesFood(msg.text)) return 'food_text';
    
    // Маркеры добавок
    if (matchesSupplements(msg.text)) return 'supplement';
    
    // Маркеры рецептов
    if (matchesRecipe(msg.text)) return 'recipe_query';
    
    // Маркеры терапии
    if (matchesTherapy(msg.text)) return 'journal_therapy';
    
    // Длинный свободный текст → дневник
    if (msg.text.length > 100 && !hasSpecificMarkers(msg.text)) return 'journal_free';
    
    // Если непонятно — спрашиваем
    return 'ask_clarification';
  }
}
```

### 4.3 Контекст предыдущих сообщений

Бот помнит последние 2-3 сообщения для контекста:
- Если предыдущее сообщение — фото еды, а следующее — "и ещё чай с печеньем" → это добавка к тому же приёму пищи
- Если бот спросил "чем занимался с 15 до 17?" → следующий ответ идёт в `time_log`
- Если была серия мыслей → всё в один `journal_entry`

## 5. Пользовательские истории

### Эпик 1: Базовая инфраструктура

**US-1.2 Supabase подключён**
> БД создана, схема применена, бот пишет и читает данные.

**US-1.3 Open Wearables развёрнут**
> Docker на Render, companion app на iPhone, данные Apple Health доступны через API.

### Эпик 2: Контекстный роутинг

**US-2.1 Классификация фото**
> Отправляю фото без caption — бот сам определяет, что это: еда, блокнот (утро/вечер), скриншот календаря. Через Claude Vision.

Требования:
- Один вызов Claude Vision для классификации + обработки одновременно
- Промпт: "Определи что на фото: еда, рукописный шаблон с чекбоксами (утро/вечер), скриншот календаря, или другое. Затем обработай соответственно."
- Если не уверен → inline-кнопки

**US-2.2 Классификация текста**
> Пишу свободный текст — бот определяет из контекста, что это: описание еды, поток мыслей, ответ на его вопрос, запрос рецепта.

Требования:
- Паттерны + длина текста + контекст предыдущих сообщений
- Длинный текст без маркеров еды → дневник
- Маркеры "психолог", "терапия", "сессия" → journal_therapy
- "Съел", "на обед" → еда
- "Что приготовить", "рецепт" → рецепт

**US-2.3 Контекст диалога**
> Бот помнит предыдущие 2-3 сообщения и использует их для классификации.

Требования:
- Хранение последних 3 сообщений в памяти (Redis-подобно, или просто переменная)
- Если бот задал вопрос → следующий ответ относится к этому вопросу
- "И ещё..." после фото еды → добавка к тому же приёму

### Эпик 3: Распознавание еды → КБЖУ

**US-3.1 Фото еды → КБЖУ**
> Отправляю фото — получаю калории, белки, жиры, углеводы.

Требования:
- Claude Vision API (Sonnet 4.6) — выше качество распознавания русских блюд
- Ответ: "Борщ со сметаной: ~320 ккал, Б:15 Ж:18 У:24 ✅"
- Inline-кнопки: 🌅 Завтрак | 🌞 Обед | 🌙 Ужин | 🍎 Перекус
- → `nutrition_log`

**US-3.2 Текст о еде → КБЖУ**
> "Куриная грудка 200г + рис 150г" → КБЖУ.

Требования:
- Claude API → расчёт → `nutrition_log`

### Эпик 4: Распознавание блокнота

**US-4.1 Фото утреннего блокнота**
> Фотографирую утренний блокнот → бот парсит, записывает, отмечает привычки.

Требования:
- Claude Vision (Sonnet) — рукописный текст на русском
- Парсинг всех секций: чекбоксы, метрики, текстовые поля, вещества
- → `notebook_entry` + `substance_log` + TickTick habits
- Ответ: "Утро ✅ Сон: 7ч (7/10). Тело: 6. Привычки: 4/5."

**US-4.2 Фото вечернего блокнота**
> Аналогично, вечерняя секция.

**US-4.3 Приоритет сна**
> Блокнот > Watch. Если в блокноте нет данных сна → Open Wearables API.

### Эпик 5: Дневник

**US-5.1 Свободный текст → дневник**
> Пишу поток мыслей в бот — он сохраняет как запись дневника.

Требования:
- Текст > 100 символов без маркеров еды → `journal_entry` (тип: free)
- Бот подтверждает: "Записал 💭" (минимальный ответ, не перебивать поток)
- НЕ анализирует, НЕ комментирует, НЕ даёт советов. Просто фиксирует.
- Timestamp + текст

**US-5.2 Записи с психолога**
> После сессии с психологом пишу ключевые мысли / инсайты.

Требования:
- Маркеры: "психолог", "терапия", "сессия", "с психолога"
- → `journal_entry` (тип: therapy)
- Бот: "Записал сессию 🧠" (минимальный ответ)
- НЕ интерпретирует, НЕ анализирует содержание
- Доступно для Claude Cowork при еженедельном/ежемесячном обзоре (если пользователь явно разрешит)

**US-5.3 Просмотр дневника**
> "Что я писал на этой неделе?" → бот показывает записи.

Требования:
- /journal или "дневник" → последние N записей
- Фильтр по типу: free / therapy
- Краткие превью (первые 100 символов)

**US-5.4 Извлечение задач из дневника → TickTick**
> Пишу в дневник поток мыслей — если там есть задача, бот сам создаёт её в TickTick inbox.

Требования:
- После сохранения `journal_entry` — Claude (Haiku) анализирует текст на наличие задач
- Промпт: "Есть ли в этом тексте задачи — конкретные действия, которые автор хочет или должен сделать? Верни JSON: [{task, urgency}] или пустой массив."
- Если задачи найдены → создание через TickTick API (`POST /open/v1/task`) в проект Inbox
- Бот дополняет подтверждение: "Записал 💭 + 📋 Задача: «позвонить маме» → inbox"
- Пользователь может inline-кнопкой отменить создание задачи (❌ Не нужно)
- Извлекает задачи из обоих типов: free и therapy (с сессий часто выходят конкретные задачи)
- Порог: задача должна быть конкретным действием, а не абстрактным желанием ("хочу быть спокойнее" — нет, "записаться к стоматологу" — да)

### Эпик 6: Аналитика распорядка дня (без слепых зон)

**US-6.1 Автоматический сбор из календарей**
> Система читает оба календаря и записывает блоки времени.

Требования:
- Google Calendar API (личный) → события → `time_log`
- Рабочий календарь (API/ICS/скриншот) → `time_log`
- Каждое событие: start_time, end_time, title, category (work/personal), source

**US-6.2 Обнаружение слепых зон**
> Если в расписании есть промежуток > 1 часа без записи, бот спрашивает.

Требования:
- Cron: 3 раза в день (11:00, 16:00, 21:00) — проверка time_log на пробелы
- Если пробел > 60 мин: "С 14:00 до 16:30 — нет записи. Чем занимался?"
- Inline-кнопки для быстрого ответа:
  🏠 Дома/отдых | 💻 Работа | 🚶 Прогулка | 🍽 Еда | 😴 Сон/отдых | ✏️ Своё
- Ответ → `time_log`

**US-6.3 Ручной ввод активности**
> "С 15 до 17 работал над PRD" → запись в time_log.

Требования:
- Парсинг: "с X до Y [описание]" → start/end/description
- → `time_log` (source: manual)

**US-6.4 Дневной таймлайн**
> "Чем я сегодня занимался?" → полный таймлайн дня.

Требования:
- Команда /day или "мой день"
- Сборка из `time_log`: хронологический список
- Подсветка пробелов (если остались)
- Процент дня по категориям: работа X%, отдых Y%, ...

**US-6.5 Недельная аналитика времени**
> В еженедельном отчёте Claude — разбивка по категориям.

Требования:
- Входит в еженедельный отчёт (Claude Cowork)
- "На этой неделе: работа 42ч, сон 49ч, отдых 12ч, прогулки 5ч, пробелы 3ч"
- Тренды: "Работа +4ч vs прошлая неделя. Прогулки -2ч."
- Сверка с целями из TickTick

### Эпик 7: Рецепты (read-only в боте)

**US-7.1 Наполнение базы рецептов (отдельная сессия)**
> Рецепты собираются в отдельной сессии Claude Cowork: парсинг с портала → Supabase.

Требования:
- Отдельная задача (не в боте): Claude Cowork / Code парсит рецепты с выбранного портала
- Структура: название, ингредиенты, шаги, время, порции, КБЖУ
- Заливка в `recipe` (Supabase)
- Периодическое обновление — тоже через Cowork-сессию

**US-7.2 Запрос рецепта через бот**
> "Что приготовить?" или "рецепт с курицей" → бот выдаёт из базы.

Требования:
- Поиск по тегам, ингредиентам, времени готовки
- Ответ: название, время, КБЖУ, краткие шаги
- Если несколько вариантов → топ-3

**US-7.3 Замена рецепта**
> "Не хочу это, что ещё?" → бот предлагает альтернативу.

Требования:
- Исключает предложенный рецепт
- Предлагает следующий по релевантности
- Учитывает ротацию (не предлагать то, что готовил вчера)

**US-7.4 Рекомендация в контексте дня**
> Обеденный/вечерний чекпоинт включает рекомендацию рецепта.

Требования:
- Учитывает: сколько КБЖУ уже съедено, время до следующего события в календаре
- "Обед: Куриный суп (30 мин, 280 ккал). Окно до встречи — 2ч, успеваешь."

### Эпик 10: Cron-напоминания

**US-10.1 Утренний брифинг (07:00)**
> Сон + календари + задачи TickTick + рекомендация рецепта + "не забудь блокнот".

**US-10.2 Обеденный чекпоинт (12:30)**
> Проверка еды + шаги + рецепт.

**US-10.3 Вечерний чекпоинт (19:00)**
> КБЖУ итого + шаги + добавки + рецепт ужина.

**US-10.4 Wind-down (22:00)**
> Напоминание о сне + вечерний блокнот.

**US-10.5 Проверка слепых зон (11:00, 16:00, 21:00)**
> Пробелы в time_log → вопрос в Telegram.

**US-10.6 Дневная сводка (23:30)**
> Агрегация всего → `daily_summary`. Тихо, без сообщения.

**US-10.7 Напоминание о еде (13:00, 15:00, 20:00)**
> Если прошло время приёма пищи, а записи нет — бот мягко напоминает поесть.

Требования:
- Cron-проверка в 13:00, 15:00, 20:00 (после типичного завтрака, обеда, ужина)
- Логика: проверяет `nutrition_log` за текущий день (count записей) + `time_log` на событие с категорией 'food'
- 13:00 — если 0 записей в nutrition_log → "Ты ещё не завтракал? 🍳 Не забудь поесть."
- 15:00 — если ≤1 запись → "Обедал? Если да — скинь фото или опиши."
- 20:00 — если ≤2 записи → "Про ужин не забыл?"
- Если есть событие 'food' в time_log за соответствующий период — пропускает напоминание (человек ел, но не логировал)
- Не напоминать, если пользователь в последние 30 мин отправлял фото еды (уже залогировал)
- Inline-кнопки: 🍽 Уже ел (→ запрос описания) | ⏭ Пропускаю | 📸 Сейчас скину фото

### Эпик 11: Claude Cowork — аналитика и наставник

**US-11.1 Еженедельный отчёт**
> Воскресенье: тренды, корреляции, аналитика времени, рекомендации.

Содержание:
- КБЖУ средние, тренды
- Сон: среднее, качество, корреляции с веществами
- Привычки: % выполнения, что просело
- Время: разбивка по категориям, тренды, слепые зоны
- Вещества: частота, паттерны (без оценок)
- Дневник: ключевые темы (если пользователь разрешил)
- Рекомендации на следующую неделю

**US-11.2 Ежемесячный наставник**
> Глубокая сессия: ревью курса, цели TickTick, паттерны из дневника и терапии (с разрешения).

### Эпик 12: Backbone рутины (тестируемый распорядок)

Концепция: пользователь описывает текстом текущий вариант рутины (распорядок дня), загружает его в систему. Бот знает этот backbone и использует при брифингах. Если что-то не получается — пользователь пишет в дневник, потом в Cowork-сессии вместе корректируют backbone.

**US-12.1 Загрузка backbone рутины**
> Пишу текстом свой текущий распорядок — бот сохраняет как активный backbone.

Требования:
- Текстовый формат (свободная форма или структурированный)
- Пример: "7:00 подъём, 7:15 блокнот, 7:30 завтрак, 8:00 work block 1, 12:30 обед, 13:00 прогулка, 14:00 work block 2, 18:00 спорт, 19:00 ужин, 21:00 вечерний блокнот, 22:30 сон"
- → `routine_config` (is_active = true), предыдущий → is_active = false
- Бот: "Backbone обновлён ✅ Версия #3"
- Команда: "рутина" / /routine → показать текущий backbone

**US-12.2 Бот учитывает backbone в брифингах**
> Утренний брифинг включает backbone дня и показывает, как реальное расписание соотносится с планом.

Требования:
- Утренний брифинг (US-10.1): "По backbone: 7:00 подъём → блокнот → завтрак. Первая встреча в 9:00."
- Cron-чекпоинты сравнивают backbone vs реальность (time_log)
- Если отклонение > 1 часа → мягкое наблюдение: "По плану сейчас прогулка, но ты на встрече — ок, корректируем."
- Не навязывает, не ругает. Факт + мягкий вопрос.

**US-12.3 Фидбэк через дневник → итерация backbone**
> Пишу в дневник "утром не успеваю делать блокнот до завтрака" — это фидбэк по backbone.

Требования:
- Бот НЕ сам корректирует backbone (слишком ответственное решение)
- Дневниковые записи с фидбэком по рутине доступны для Claude Cowork
- В еженедельном отчёте (US-11.1): раздел "Backbone рутины" — что получалось, что нет, паттерны отклонений
- В Cowork-сессии: пользователь + Claude вместе решают, как скорректировать
- После корректировки → пользователь загружает новый backbone (US-12.1)

**US-12.4 История версий backbone**
> Могу посмотреть предыдущие версии рутины.

Требования:
- /routine history → список версий с датами
- Показать конкретную версию по номеру
- Полезно для Cowork-анализа: "С версией 2 ты спал на 40 мин больше"

## 10. Аудит готовности к разработке

Ниже — пробелы и области, требующие уточнения перед тем, как передавать PRD в работу.

### 🔴 Критические (блокируют старт разработки)

**9.1 Безопасность Telegram-бота**
Бот принимает сообщения от любого пользователя. Нужен `chat_id` whitelist — только твой аккаунт.
*Решение:* env `ALLOWED_CHAT_IDS`, проверка в middleware до роутинга.

**9.2 Хранение секретов и OAuth-токенов**
PRD упоминает 6+ API-ключей (Telegram, Claude, Supabase, TickTick, Google, Open Wearables), но не описывает:
- Где хранить refresh-токены TickTick и Google (в Supabase? в env?)
- Логика обновления access-токенов (TTL, автообновление)
- Что происходит при истечении токена в середине cron-джоба
*Требуется:* таблица `auth_tokens` или решение по хранению + описание refresh-flow.

**9.3 Таймзона**
Все cron-задачи указаны по часам (07:00, 12:30, 22:00), но не указана таймзона. Render Docker работает в UTC.
*Решение:* env `TZ=Asia/Yekaterinburg` (или другой) + все cron-выражения в локальном времени.

### 🟡 Важные (нужны до завершения MVP)

**9.4 Промпты Claude Vision**
PRD описывает что бот делает, но не содержит конкретных промптов для:
- Классификации фото (еда vs блокнот vs календарь)
- Парсинга еды → КБЖУ (формат ответа, handling edge cases)
- Парсинга блокнота → JSON (секции, чекбоксы, метрики)
- Извлечения задач из дневника
*Требуется:* отдельный файл `prompts.md` с шаблонами промптов и примерами ожидаемого output.

**9.5 Маппинг привычек блокнота → TickTick**
Блокнот содержит чекбоксы (зубы, зарядка, витамины и т.д.). TickTick хранит привычки с ID. Как бот узнаёт, какой чекбокс = какая привычка?
*Варианты:* конфиг-файл маппинга, или Supabase-таблица `habit_mapping`, или CLI-команда настройки.

**9.6 Формат парсинга блокнота**
Шаблон блокнота задокументирован (в памяти), но PRD не содержит JSON-схему результата парсинга. Разработчику нужно знать:
- Точная структура `notebook_entry.checkboxes` JSONB
- Точная структура `notebook_entry.metrics` JSONB
- Как обрабатывать ситуацию, когда поле не заполнено / нечитаемо
*Требуется:* пример JSON-output парсинга для утреннего и вечернего блокнота.

**9.7 Ошибки и fallback**
Не описана стратегия при:
- Claude API недоступен / 429 rate limit
- Supabase недоступен
- TickTick API ошибка (создание задачи не удалось)
- Telegram webhook не доставлен
*Минимум:* retry с exponential backoff (1, 2, 4 сек), сообщение пользователю при полном провале, логирование ошибок.

**9.8 In-memory контекст на Render**
US-2.3 требует хранения последних 3 сообщений. Render free tier перезапускает контейнер — in-memory теряется.
*Варианты:* хранить контекст в Supabase (таблица `conversation_context`), или Redis (но это не $0). Supabase добавляет ~30мс латентности, но для Telegram-бота это ок.

### 🟢 Желательно (улучшат качество, но не блокируют)

**9.9 Тестирование**
Нет стратегии тестирования. Предлагается:
- Unit-тесты: маркеры текста (matchesFood, matchesTherapy и т.д.)
- Integration: Claude Vision mock → проверка парсинга
- E2E: отправка тестового сообщения → проверка записи в Supabase
*Достаточно для MVP:* unit-тесты на роутинг + integration на парсинг.

**9.10 Логирование и мониторинг**
Нет описания. Для free tier предлагается:
- Structured JSON logs (stdout → Render logs)
- Ежедневный health-check в daily_summary (успешные/неуспешные API-вызовы)
- UptimeRobot уже есть для пинга; он же мониторит uptime.

**9.11 Мелкие проблемы в модели данных**
- `journal_entry`: trailing comma после `tags TEXT[]` — синтаксическая ошибка SQL
- `nutrition_log`: нет поля `date` (только `created_at`) — неконсистентно с другими таблицами, усложняет запросы по дате
- `supplement_log.time_of_day` vs `substance_log.time_taken` — разные имена для похожего концепта
- `daily_summary` не ссылается на `routine_config.version` — полезно для корреляций
- Нет индекса на `supplement_log(date)`
